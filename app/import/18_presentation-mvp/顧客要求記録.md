ご指摘の通り、「顧客情報(A)」「あらゆるインプットデータとその解説(B, C)」を基に、AIが「一撃HTMLプロンプト」の知識を活用し、**エージェントのように振る舞い、各ページに最適なHTMLパターンを判断・適用して全体を構成する**というコアアイデアは非常に強力です。

# 高度情報入力型 プレゼンテーション生成Webアプリ 要件定義書 (改訂版)

## 1. 概要

本システムは、ユーザーが提供する「対象顧客情報(A)」、「インプット情報のコンテキスト解説(B)」、および「プレゼンテーションの元となる多様な情報群(C)」を統合的に解析し、プロフェッショナル品質のHTMLプレゼンテーションを自動生成するWebアプリケーションである。AIエージェントが、既存の高度なHTMLテンプレート群（一撃HTMLプロンプトで定義されたもの）の構造と文脈を理解し、入力情報に応じて最適なレイアウトパターンを各ページに適用・判断することで、高品質かつカスタマイズ性の高いプレゼンテーションを効率的に作成することを目指す。

## 2. 機能要件

### 2.1. ユーザーインターフェース (UI)

*   **FR-UI-001: プロジェクト作成/管理画面**
    *   新規プレゼンテーションプロジェクトの作成機能。
    *   （将来拡張）過去プロジェクトの保存、一覧表示、再編集機能。
*   **FR-UI-002: コア情報入力インターフェース**
    *   **Aウィンドウ (対象顧客情報入力エリア):**
        *   プレゼンテーションの対象となる個人または組織の情報を自由テキスト形式で入力するエリア。
        *   入力例として「名刺情報、メール署名、企業の基本情報などを貼り付けてください」とガイド。
    *   **Bウィンドウ (インプット情報コンテキスト解説エリア):**
        *   後述のCウィンドウに入力される情報群が「何に関する情報か」「どのような背景や目的を持つか」などを自由テキスト形式で解説するエリア。
        *   入力例として「C1はX社との初回打ち合わせ議事録、C2は競合Y社の公開IR情報です。これらを基にX社への改善提案を作成します」とガイド。
    *   **Cウィンドウ群 (インプット情報入力エリア):**
        *   プレゼンテーションの元となる情報を複数入力できる動的なエリア群。
        *   各Cウィンドウ（C1, C2, C3...）は独立した大きなテキストエリア。
        *   ユーザーは「＋情報ソースを追加」ボタンでCウィンドウを増やすことができる。
        *   各Cウィンドウには、トークスクリプト、議事録、リサーチデータ、Webサイトのコピー、既存資料のテキスト、箇条書きメモ、スライド構成案の断片（例:「この部分は課題提起で、Aパターンを使ってほしい」など）を自由に入力可能。
*   **FR-UI-003: プレゼンテーション基本設定フォーム**
    *   プレゼンテーション全体のタイトル、文書種類、機密レベル、作成者名、日付など、メタデータ的な情報を入力する最小限のフォーム。
*   **FR-UI-004: 生成オプション設定 (簡易)**
    *   （初期MVPでは省略可）全体のトーン（例: フォーマル、クリエイティブ）、強調したいポイントなどをキーワードで指定できる簡易的なオプション。
*   **FR-UI-005: 生成実行ボタン**
    *   入力されたA, B, Cの情報および基本設定に基づいて、HTMLプレゼンテーションの生成を開始するボタン。
*   **FR-UI-006: 進捗表示・フィードバックエリア**
    *   生成処理の現在のステータス（例: 「顧客情報を分析中...」「スライド構成を検討中...」「スライド1/Nを生成中...」）をリアルタイムに近い形で表示。
    *   AIエージェントがどのような判断をしているか（例:「議事録C1から課題点を抽出し、レイアウトBでの表現を検討しています」）を適度にフィードバックすることで、ユーザーに透明性とコントロール感を提供（高度機能）。
*   **FR-UI-007: プレビュー機能**
    *   生成されたスライドを順次、または全スライド生成後にプレビュー表示する機能。
    *   スライド単位での簡易的な修正指示（例：「このスライドのレイアウトをCパターンに変えてほしい」といったフィードバックを送信し、再生成を促す機能 – 高度機能）。
*   **FR-UI-008: ダウンロード機能**
    *   生成された完全なHTMLファイルをダウンロードする機能。

### 2.2. プレゼンテーション生成エンジン (バックエンド) - AIエージェントシステム

*   **FR-BE-001: 入力情報統合解析モジュール (エージェント1 - 情報分析官)**
    *   A, B, Cウィンドウから入力された全てのテキスト情報を統合的に解析。
    *   Aから対象顧客の特性、ニーズ、課題を抽出。
    *   Bからプレゼンテーションの目的、主要メッセージ、背景を理解。
    *   Cから具体的なコンテンツ要素、データ、ユーザーの意図（レイアウト示唆含む）を抽出・構造化。
    *   情報間の関連性や重要度を評価。
*   **FR-BE-002: プレゼンテーション構成設計モジュール (エージェント2 - ストーリーテラー/構成作家)**
    *   解析された情報と、「一撃HTMLプロンプト」内に含まれる多様なスライド構成・レイアウトパターンの知識ベースを参照。
    *   プレゼンテーション全体の論理的な流れ（序論、本論、結論、各部の詳細項目）を設計。
    *   各スライドに割り当てるべきコンテンツ要素と、それに最適なHTMLレイアウトパターン（A～J）を判断・提案。
        *   Cウィンドウ内にユーザーからのレイアウト指定があれば優先的に考慮。
        *   指定がない場合は、コンテンツの特性（テキスト量、データ種別、図表の必要性など）と「一撃HTMLプロンプト」の成功事例から類推して判断。
    *   ページネーションとページ番号の割り振り。
*   **FR-BE-003: スライドコンテンツ生成モジュール (エージェント3 - コピーライター/デザイナー補助)**
    *   設計されたスライド構成と選択されたレイアウトパターンに基づき、各スライドの具体的なコンテンツ（テキスト、Mermaid定義、Chart.js用データ構造など）を生成。
    *   「一撃HTMLプロンプト」内の各レイアウトテンプレートに対応する変数を埋める形でコンテンツを出力。
    *   文体、トーンはA, Bの情報から推測される顧客や目的に合わせて調整。
*   **FR-BE-004: HTML組立・出力モジュール**
    *   生成された各スライドコンテンツを、対応するHTMLテンプレートパーツ（cover.html, detail_pattern_A.htmlなど）に適用。
    *   全てのHTMLパーツを`base.html`に統合し、単一の完全なHTMLファイルを生成。
    *   CSSスタイル（印刷対応含む）およびJavaScriptを適切に組み込む。
*   **FR-BE-005: LLM連携管理**
    *   上記各エージェントモジュールとLLM API間の効率的かつ信頼性の高い通信を管理。
    *   プロンプトの動的構築、コンテキスト長管理、APIエラーハンドリング、リトライ処理。
*   **FR-BE-006: ナレッジベース管理**
    *   「一撃HTMLプロンプト」の内容（特に各レイアウトパターンとそれに適したコンテンツの特性、変数定義など）をLLMが参照しやすい形でナレッジベース化。

## 3. 非機能要件

*   **NFR-PF-001: パフォーマンス**
    *   コア情報入力から初回プレビュー（または全生成完了）までの目標時間：入力情報量と複雑性に依存するが、平均的なケース（例: Cウィンドウ3つ、合計1万字程度）で X 分以内（要件定義で具体化）。
    *   UIの応答性は高く、テキスト入力や追加操作はストレスなく行えること。
*   **NFR-SC-001: スケーラビリティ**
    *   バックエンドのAIエージェント処理（特にLLM呼び出し）は、並列処理や非同期処理を活用し、負荷に応じてスケールアウト可能な設計。
*   **NFR-US-001: ユーザビリティ**
    *   A, B, Cの入力インターフェースはシンプルで、ユーザーが迷わず情報を入力できること。
    *   AIの処理プロセスがある程度可視化され、ブラックボックス感を軽減すること。
    *   生成AIの利用経験が浅いユーザーでも、「とりあえず情報を入れれば形になる」という期待感を持てること。
*   **NFR-RL-001: 信頼性・一貫性**
    *   AIエージェントによるレイアウト判断やコンテンツ生成は、入力に対してある程度一貫性のある結果を出すこと（ただし、生成AIの特性上、完全な決定論性は求めない）。
    *   生成されるHTMLは、指定された品質基準（16:9比率、A4横印刷対応など）を常に満たすこと。
*   **NFR-FL-001: 柔軟性**
    *   Cウィンドウに入力される情報の種類や形式に対して、システムがある程度柔軟に対応できること。
*   **NFR-MN-001: メンテナンス性**
    *   AIエージェントの各モジュールは論理的に分離され、プロンプトやロジックの改善が容易であること。
    *   HTMLテンプレート群は独立して管理・更新可能であること。
*   **NFR-ST-001: セキュリティ**
    *   ユーザー入力データ（特に顧客情報や機密情報を含む可能性のあるCウィンドウの内容）は適切に保護されること。
    *   LLM APIキー等の認証情報は安全に管理されること。

*   業界や目的に特化した知識ベースの追加と、それに基づく提案力の強化。

# 高度情報入力型 プレゼンテーション生成Webアプリ 技術定義書 (改訂版)

## 1. システムアーキテクチャ概要

本システムは、クライアントサイド（フロントエンドSPA）とサーバーサイド（バックエンドAPIおよびAIエージェントシステム）から構成される。ユーザーはブラウザを通じてコア情報を入力し、バックエンドのAIエージェントシステムがLLMと連携してHTMLプレゼンテーションを生成する。AIエージェントは複数の論理モジュール（情報分析官、構成作家、コピーライター補助）として振る舞い、それぞれが連携して処理を進める。

## 2. 技術スタック

### 2.1. フロントエンド

*   **言語:** TypeScript
*   **フレームワーク/ライブラリ:** React.js (Next.js) または Vue.js (Nuxt.js)
    *   理由: 大規模なテキスト入力エリアや動的なコンポーネント管理、状態管理に適している。Next.js/Nuxt.jsは開発効率とパフォーマンスに貢献。
*   **UIライブラリ/CSS:**
    *   Tailwind CSS + DaisyUI (既存資産との親和性、迅速なUI構築)
    *   リッチテキストエディタライブラリ (Cウィンドウ群に高度な編集機能を提供する場合。例: TipTap, Quill, Slate) - 初期はプレーンなtextareaでも可。
*   **状態管理:** Redux Toolkit (React), Pinia (Vue), または Zustand (軽量)
*   **HTTPクライアント:** Axios または Fetch API
*   **WebSocketクライアント:** Socket.IO Client または標準WebSocket API (進捗表示のため)

### 2.2. バックエンド (APIサーバー & タスクオーケストレーター)

*   **言語:** Python
    *   理由: LLMライブラリ、データ処理ライブラリが豊富。AIエージェントロジックの実装に適している。
*   **フレームワーク:** FastAPI
    *   理由: 高パフォーマンス、非同期処理対応、型ヒントによる自動バリデーションとAPIドキュメント生成。
*   **タスクキュー/メッセージブローカー (AIエージェント間の連携、非同期処理):** Celery with Redis or RabbitMQ
    *   理由: Pythonエコシステムでの実績、複雑なワークフロー管理が可能。
*   **データベース (ジョブ管理、ナレッジベースのインデックス等):** PostgreSQL または MongoDB
    *   理由: 信頼性、スケーラビリティ。ナレッジベースの特性に応じて選択。
*   **HTMLテンプレートエンジン (サーバーサイドでの最終組立):** Jinja2

### 2.3. AIエージェントシステム

*   **LLM連携:**
    *   OpenAI API (GPT-4, GPT-3.5-turbo), Google Gemini API, Anthropic Claude API など、複数の高性能モデルをタスクに応じて使い分けることも視野に入れる。
    *   公式SDK (openai-python, google-generativeai, anthropic)
    *   **フレームワーク (推奨):** LangChain または LlamaIndex
        *   理由: エージェント構築、プロンプト管理、LLMとのインタラクション抽象化、外部ツール連携、CoT (Chain of Thought) やReAct (Reasoning and Acting) パターンの実装を支援。ナレッジベース連携（RAG - Retrieval Augmented Generation）にも有用。
*   **ナレッジベース構築・参照:**
    *   「一撃HTMLプロンプト」のHTML構造、レイアウトパターンの特性、変数定義、成功例などを構造化データ（JSON, YAML）やベクトルデータベース（FAISS, Pinecone, Weaviateなど）に格納。
    *   LlamaIndexやLangChainの機能を利用して、関連情報を効率的に検索・LLMのコンテキストに注入（RAG）。
*   **プロンプトエンジニアリング:** 各AIエージェントモジュール（情報分析官、構成作家、コピーライター補助）の役割に応じた高度なプロンプトチェーンを設計。

### 2.4. 開発・運用ツール

*   **バージョン管理:** Git, GitHub/GitLab
*   **コンテナ化:** Docker, Docker Compose
*   **オーケストレーション (将来):** Kubernetes
*   **CI/CD:** GitHub Actions, GitLab CI/CD
*   **監視・ロギング:** ELK Stack, Prometheus, Grafana, Sentry
*   **APIドキュメンテーション:** FastAPIによる自動生成 (Swagger UI, ReDoc)

## 3. データフローとAIエージェント処理ロジック

1.  **ユーザー入力 (フロントエンド):** ユーザーがA, B, Cウィンドウに情報を入力し、基本設定と共にバックエンドAPIに送信。
2.  **ジョブ登録 (バックエンドAPI):**
    *   入力データを検証し、一意のジョブIDを発行。
    *   生成タスクをタスクオーケストレーター（Celery等で実装）に登録。
3.  **タスクオーケストレーション開始:**
    *   **情報分析官エージェント (Agent1):**
        *   入力されたA, B, C情報をLLMに渡し、顧客プロファイル、プレゼン目的、主要テーマ、利用可能なコンテンツ要素、ユーザーの隠れた意図（レイアウト示唆など）を抽出・構造化するよう指示。
        *   結果をオーケストレーターに返す。
    *   **構成作家エージェント (Agent2):**
        *   Agent1の分析結果と、ナレッジベース（「一撃HTMLプロンプト」の構造情報、レイアウトパターン特性）を参照。
        *   LLMに対し、最適なプレゼンテーション構成（全体の流れ、各スライドのトピック、推奨レイアウトパターンA～J）を提案するよう指示。ユーザーがCで示したレイアウト希望も考慮。
        *   結果（スライド構成案と各スライドのレイアウト指示）をオーケストレーターに返す。
    *   **コピーライター補助エージェント (Agent3 - スライドごとに並列実行可能):**
        *   オーケストレーターから、個々のスライドに割り当てられたトピック、関連情報（Agent1の分析結果から抜粋）、および決定されたレイアウトパターンを受け取る。
        *   HTMLテンプレート群（各レイアウトパターン用）とLLMを参照し、具体的なスライドコンテンツ（テキスト、Mermaidコード、Chart.jsデータ等、テンプレートの変数を埋める形）を生成するよう指示。
        *   生成されたスライドコンテンツパーツをオーケストレーターに返す。
4.  **HTML組立 (HTMLBuilder):**
    *   オーケストレーターがAgent3から全スライドパーツを受け取った後、HTML組立モジュールに指示。
    *   各パーツを`base.html`に統合し、最終的なHTMLファイルを生成。
5.  **結果通知・配信 (バックエンドAPI):**
    *   完成したHTMLファイルへのアクセス方法（ダウンロードURLや直接配信）をフロントエンドに通知。
    *   処理中、WebSocketを通じて各エージェントの処理状況や中間的な判断をフロントエンドにフィードバック。
6.  **プレビュー/ダウンロード (フロントエンド):** ユーザーは進捗を確認しつつ、完成したHTMLをプレビューまたはダウンロード。

## 4. 「エージェントティックな紐付け」の実現アプローチ

*   **マルチエージェントシステム:** 上述の通り、情報分析、構成設計、コンテンツ生成の役割を論理的に分離したエージェント群としてLLMを活用。
*   **プロンプトチェーンと状態管理:** LangChainのようなフレームワークを利用し、各エージェントの出力を次のエージェントの入力とするプロンプトチェーンを構築。タスクオーケストレーターがエージェント間の状態（生成された中間成果物）を管理。
*   **RAG (Retrieval Augmented Generation):** 「一撃HTMLプロンプト」の知識（レイアウトパターンと適切なコンテンツの関連性など）をベクトルDB等に格納し、構成作家エージェント(Agent2)がレイアウト判断する際に最も関連性の高い情報を検索し、LLMのプロンプトに付加する。これにより、LLMは固定の知識だけでなく、具体的なテンプレート群の特性に基づいた判断が可能になる。
*   **Chain of Thought (CoT) / ReAct プロンプティング:** 特に構成作家エージェント(Agent2)において、単に結果を出力させるだけでなく、「なぜこの構成なのか」「なぜこのレイアウトパターンが適切なのか」という思考プロセスをLLMに出力させ、その思考プロセス自体を判断材料の一部とする。
*   **フィードバックループ (高度):** 初期生成後、ユーザーが特定スライドのレイアウト変更を指示した場合、そのフィードバックを構成作家エージェント(Agent2)またはコピーライター補助エージェント(Agent3)に渡し、該当部分を再生成させるループを組み込む。

## 5. パフォーマンスとスケーラビリティ戦略

*   **非同期・並列処理:** Celery等を利用し、LLM呼び出しを伴う重い処理（特にAgent3の各スライド生成）を非同期かつ並列に実行。
*   **コンテキスト最適化:** 各LLM呼び出しにおいて、必要最小限かつ効果的なコンテキスト情報を渡すことで、処理速度と精度を向上、コストを抑制。
*   **ストリーミング応答:** LLMの応答をストリーミングで受け取り、フロントエンドの進捗表示やプレビューに逐次反映させることで体感速度を向上。
*   **インフラのスケーリング:** バックエンドAPIサーバー、Celeryワーカー、データベースは負荷に応じて独立してスケール可能な構成（コンテナ化とオーケストレーションツール利用）。

## 6. セキュリティ考慮事項

*   **入力サニタイズ:** ユーザー入力値（特にCウィンドウの自由テキスト）がLLMプロンプトやHTML生成に悪影響を与えないよう、適切なサニタイズ処理やコンテキスト分離を検討。
*   **LLMプロンプトインジェクション対策:** LangChainなどのフレームワークが提供する対策や、厳格な入力制御。
*   **API認証・認可:** バックエンドAPIへのアクセス制御。
*   **機密情報管理:** LLM APIキー、DB認証情報などの安全な管理（環境変数、シークレットマネージャー）。
*   **データ保護:** ユーザーが入力する可能性のある機密情報（顧客情報、議事録など）の取り扱いポリシーを明確化し、保存する場合は暗号化等の対策を講じる。
